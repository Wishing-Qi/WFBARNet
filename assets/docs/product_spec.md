WFBARNet 产品规范

输入输出规范与功能规范

系统输入支持本地视频、摄像头实时画面和批量视频文件夹三种方式。本地视频用于单段训练或比赛片段分析，摄像头用于实时推理，批量文件夹用于对多个回合片段统一处理并支持切换查看。系统输出包括三类结果：一是可视化结果，包括主画面中的球轨迹、球点、人体骨架、球场线、事件提示，以及标准球场视图中的球员投影和热力图；二是统计结果，包括回合时长、击球次数、平均击球间隔、落点次数、出画次数、双方跑动距离、平均速度、最大速度、启动次数、急停次数、高强度移动次数、前中后场击球分布、运动强度评分和数据可靠性指标；三是结构化结果，包括批量分析 JSON、汇总 CSV、逐帧 JSONL 和轨迹调试 CSV。

系统功能围绕球、人、场、回合四条主线设计。视频管理负责模式切换、选择输入源、预览、进度控制、停止和重置；轨迹跟踪负责逐帧定位羽毛球并输出球点坐标、可见性和置信度，同时进行短时丢失修复和误检抑制；姿态估计负责识别双人关键点和人体框，稳定区分上方球员与下方球员，并提取脚下锚点；球场检测负责识别球场线并估计图像到标准场地的映射关系；事件识别负责生成击球、落点和出画事件；BST 动作识别在有权重时输出动作类别和置信度；统计模块负责把轨迹、姿态、球场和事件结果聚合为回合记录；导出和设置模块负责结果保存、模型路径配置、模块开关和调试日志管理。

核心流程与页面逻辑

用户在左侧输入区选择视频、摄像头或批量模式后启动分析，系统持续读取视频帧并分发到轨迹、姿态和球场检测模块。轨迹分支输出球点位置、可见性和分数，姿态分支输出球员框、关键点和身份，球场分支输出球场有效性、球场线和映射矩阵。系统随后汇总三路结果，完成事件识别、球员投影、跑动累计、击球归属估计和回合数据更新。回合开始优先取第一段稳定球飞行或首个击球事件，回合结束取可信落点事件，出画事件只计数不结束回合，回合结束后当前统计冻结。当前系统默认一个视频或片段对应一个统计单元，不自动切分整场比赛中的多个回合。

页面采用单窗口多区域结构。顶部区域负责开始分析、重置、主题切换、状态提示和进度显示；左侧区域负责输入源、模式和片段切换；中央区域负责视频预览和分析叠加显示；右侧标准球场区域负责球员投影、球点投影和热力图。结果区按标签页组织，概览页显示动作时序和关键状态，数据页显示回合汇总与击球详情，统计页显示击球类型累计结果，姿态页强调双人运动信息，日志页显示系统与事件记录，设置页负责模型和调试参数。模式切换后控件状态需要同步更新，分析中持续刷新结果，停止后保留当前结果，重置后清空工作区、日志和统计内容。

模块设计与接口设计

产品按界面层、控制层、算法层、后处理层和输出层划分。`apps/pyqt6/views` 负责主窗口、视频面板、热力图和结果表格，`apps/pyqt6/controllers` 负责模式切换、任务调度、线程控制、界面刷新和导出触发，`src/models` 负责轨迹模型、姿态模型和 BST 动作识别，`src/court` 负责球场识别和场地映射，`src/postprocess` 负责轨迹滤波、事件识别、姿态稳定、距离累计和回合统计，`src/utils` 负责可视化叠加、逐帧记录和结果导出。当前实现上，轨迹采用 TrackNetV3 风格链路，姿态以 YOLO pose 为主，球场检测采用多后端球场线识别与单应性估计，动作分类采用可选 BST 时序模型。

模块之间通过统一结果对象衔接。轨迹模块输出球点坐标、可见状态、置信度和候选调试信息；姿态模块输出人体框、关键点、关键点分数和球员身份；球场模块输出球场有效性、球场线、投影结果和映射矩阵；事件模块输出事件类型、帧号、时间戳、球点位置、触发规则和置信度；统计模块输出 `rally_record`，其中 `summary` 用于页面汇总和批量导出，`details` 用于击球明细和回合详情。视频预览与实时推理共用视频显示接口，数据页与批量导出共用回合记录接口，标准球场视图与统计功能共用球员投影接口；批量模式下每个片段生成独立回合记录，由控制层按片段 ID 维护索引并支持页面切换和统一导出。

关键指标、验收标准与约束

关键指标分为回合指标、运动指标、空间指标和质量指标。回合指标包括回合时长、击球次数、平均击球间隔、落点次数和出画次数；运动指标包括双方累计跑动距离、平均速度、最大速度、启动次数、急停次数、高强度移动次数、最长连续移动距离和运动强度评分；空间指标包括前中后场击球次数、平均站位深度、前后移动比例、左右移动比例和热力图；质量指标包括球可见率、姿态有效率、球场有效率和平均球置信度；BST 可用时增加动作类型分布、动作识别置信度和 Top-K 候选结果。展示时需要结合可靠性解释，球场有效率低时弱化空间指标，姿态有效率低时弱化运动指标，球可见率低时弱化击球和落点相关指标。

验收以完整分析闭环为准。系统应至少在一种输入模式下稳定完成分析，主画面能够显示球轨迹、球点、人体骨架和球场线等主要结果，标准球场视图能够展示球员投影或热力图，数据页能够显示回合汇总和击球明细，批量模式能够为多个片段生成独立回合记录并导出 JSON 或 CSV。没有 BST 权重时应仍能完成基础分析链路，有 BST 权重时应能在概览或统计页展示动作分类结果。停止、重置、模式切换、片段切换和导出操作都应有清晰反馈，不能造成界面状态与结果记录错乱。

系统约束主要体现在范围和异常处理上。当前系统默认单个视频或片段对应一个统计单元，不自动切分整场比赛的多回合数据，不提供比分、胜负、制胜分、非受迫失误、发球权等高层比赛语义，也不保证真实三维球速或杀球速度等高维空间指标的准确性。短时轨迹丢失时系统应优先修复并记录出画事件，球场检测失效时允许继续显示视频结果但降低场地统计可信度，姿态缺失时暂停对应球员距离和速度累计，无 BST 权重时仅保留击球候选和轨迹事件展示。整体上，系统定位为训练分析和算法验证工具，所有量化结果都需要结合视频质量、拍摄角度、遮挡情况和模型匹配程度解释。
