## 2. 项目背景与问题

**核心观点**：羽毛球训练和比赛复盘需要可量化、可回放、可解释的数据支撑。

**页面内容**

- 传统人工复盘依赖经验，难以稳定提取球速、站位、移动距离和击球分布。
- 羽毛球目标小、速度快、遮挡多，对视频分析算法提出较高要求。
- 单纯检测球或人不够，需要同时理解“球、球员、球场”的空间关系。
- 项目目标是构建可本地运行、可调试、可扩展的辅助分析系统。

**建议视觉**

- 左侧列出痛点，右侧展示项目对应能力。
- 可用三对象结构图：羽毛球、球员、球场。

## 3. 项目定位与总体目标

**核心观点**：WFBARNet 不是自动裁判系统，而是训练复盘和算法验证工具。

**页面内容**

- 输入：本地视频、摄像头实时流或批量视频帧。
- 输出：球轨迹、人体骨架、球场投影、击球/落点/出画事件、回合统计和调试日志。
- 使用场景：训练复盘、技战术观察、批量数据整理、后续模型实验。
- 系统定位：本地化、可解释、可调试、可扩展。

**建议视觉**

- 采用“输入-处理-输出-应用”的四段式图。
- 在输出区放置 JSON/CSV/视频可视化/热力图等标签。

## 4. 系统总体架构

**核心观点**：系统由 PyQt6 应用层、视觉推理层、后处理统计层和数据导出层组成。

**页面内容**

- 应用层：PyQt6 桌面界面、视频播放、摄像头实时分析、数据页展示。
- 推理层：TrackBranch、PoseBranch、CourtLineDetector、可选 BSTStrokeRecognizer。
- 后处理层：轨迹滤波、姿态稳定、事件检测、球员距离累计、回合统计。
- 输出层：画面叠加、标准球场视图、JSON/CSV/NPY/JSONL 调试文件。

**建议视觉**

```text
视频/摄像头
  -> TrackNetV3 轨迹分支
  -> YOLO Pose 姿态分支
  -> OpenCV/MonoTrack 球场分支
  -> 轨迹事件 + 回合统计 + UI 可视化 + 数据导出
```

## 5. 核心流程：从视频帧到分析结果

**核心观点**：每一帧经过多路推理、统一结构封装，再进入事件和统计模块。

**页面内容**

- 构造 TrackNet 三帧窗口 `[prev, current, next]`。
- 球场检测服务生成 `court_prediction`。
- TrackBranch 输出羽毛球候选点。
- PoseBranch 输出人体框和关键点。
- BallTrackFilter 输出最终球点。
- RealtimeTrajectoryEventDetector 生成 `hit`、`landing`、`out_of_frame`。
- UI payload 和日志系统同步更新。

**建议视觉**

- 放一张纵向流水线图。
- 强调统一结果结构 `FrameResult(frame_id, pose, track)`。

## 6. 羽毛球轨迹跟踪模块

**核心观点**：轨迹分支解决小目标、高速运动、短时丢失和误检过滤问题。

**页面内容**

- 模型风格：TrackNetV3，输入为连续三帧，输出羽毛球热力图。
- 预处理：resize 到 `512 x 288`，BGR 转 RGB，三帧拼接成 9 通道输入。
- 候选解码：连通域分析、峰值分数、区域均值、面积和紧致度排序。
- 当前默认候选点最多保留 5 个。
- 支持 PyTorch 权重和 TensorRT `.engine` 后端。

**建议视觉**

- 展示“原始帧 -> 热力图 -> 候选点 -> 最终球点”的四步图。
- 用颜色标出最高分候选和被过滤候选。

## 7. 轨迹滤波与连续性修复

**核心观点**：后处理模块把模型候选点转换为稳定、可解释的逐帧轨迹。

**页面内容**

- 当前生产路径使用 `TrackNetV3TrajectoryFilter`。
- 可见候选点优先保留，低置信度或越界候选被过滤。
- fixed-lag 模式可对中间缺失段做线性 inpaint。
- 保留旧 `BallTrackFilter` 状态机和可插拔算法接口，便于替换滤波策略。
- 调试日志记录候选数量、输出点、动作原因、inpaint、relock 等字段。

**建议视觉**

- 展示一段轨迹：原始候选点散乱，滤波后变成连续尾迹。
- 页面角落列出典型调试字段：`action`、`reason`、`output_x/y`、`inpaint_mask`。

## 8. 球员姿态估计与稳定

**核心观点**：姿态模块让系统从“球的位置”扩展到“球员行为与运动负荷”。

**页面内容**

- 主后端：Ultralytics YOLO Pose。
- 可扩展后端：MMPose。
- 输出：人体框、关键点、关键点置信度和球员身份索引。
- 用于绘制骨架、估计脚下锚点、区分上方/下方球员。
- 结合球场位置和历史结果稳定球员身份，减少抖动和漏检影响。

**建议视觉**

- 在比赛帧上叠加人体骨架。
- 右侧展示“关键点 -> 脚下锚点 -> 场地坐标”的转换过程。

## 9. 球场线检测与场地坐标映射

**核心观点**：单应性矩阵把图像像素转换为真实场地空间，使统计指标更有意义。

**页面内容**

- 默认后端：OpenCV 球场线检测。
- 可选后端：MonoTrack 风格传统 CV 后端。
- 统一接口：`create_court_line_detector(...)`、`predict_court_lines(...)`。
- 输出：球场角点、标准模板投影、`image_to_court_h` 单应性矩阵。
- 用途：球场线叠加、球员位置投影、热力图、击球区域和移动距离统计。

**建议视觉**

- 左侧为原视频球场线，右侧为标准羽毛球场平面。
- 用箭头表示从图像坐标到厘米级场地坐标。

## 10. 轨迹事件识别

**核心观点**：系统通过规则式方法识别击球、落点和出画事件，便于解释与调试。

**页面内容**

- 主事件来源：`RealtimeTrajectoryEventDetector`。
- 事件类型：
  - `hit`：击球候选事件。
  - `landing`：落点或回合结束候选事件。
  - `out_of_frame`：出画或跟踪丢失事件。
- 主击球规则重点依赖 `vy_reversal` 和 `vx_reversal`。
- 加速度峰值、速度局部峰值等作为辅助证据，降低误检。
- 落点检测考虑速度突变、低速尾段、长时间 tracking lost 等情况。

**建议视觉**

- 在轨迹线上用红点、绿菱形、紫色叉号标记三类事件。
- 配一张规则说明小表格。

## 11. 击球动作识别与 BST 输入

**核心观点**：当提供 BST 权重时，系统可在击球事件附近进一步识别动作类别。

**页面内容**

- 击球事件触发时，截取附近时序片段。
- 输入特征包括人体姿态、羽毛球位置、球员场地位置。
- 输出动作类别、置信度、Top-K 候选和对应帧号。
- 没有 BST 权重时，系统仍能完成轨迹、姿态、球场和回合统计。
- `BSTInputBuilder` 可导出后续训练或实验所需数据。

**建议视觉**

- 展示“hit_event -> 时序片段 -> BST 分类 -> 击球类型”的流程。
- 可放击球类型分布饼图作为结果示意。

## 12. 回合统计与数据指标

**核心观点**：系统把逐帧视觉结果汇总成可用于训练复盘的回合级指标。

**页面内容**

- 回合状态：未开始、回合中、回合结束。
- 回合开始：稳定球飞行段或第一条 `hit` 事件。
- 回合结束：可信 `landing` 事件；`out_of_frame` 只计数，不单独结束回合。
- 统计指标：
  - 回合时长、击球次数、落点次数、出画次数。
  - 球员累计跑动距离、平均速度、最大速度。
  - 启动次数、急停次数、高强度移动次数。
  - 前场/中场/后场击球分布。
  - 球点、姿态、球场有效率。

**建议视觉**

- 双球员对比表。
- 回合状态时间轴。
- 击球区域三段式球场示意图。

## 16. 当前边界、风险与后续优化

**核心观点**：系统已形成完整分析链路，但准确性仍依赖视频条件、模型权重和事件规则质量。

**当前边界**

- 不是裁判级系统，不判断界内/界外、犯规、重发球等复杂比赛规则。
- 当前没有完整的多回合自动切分算法；默认把当前视频或片段作为一个统计单元。
- 回合结束依赖可信 `landing` 事件，误检会提前结束，漏检会保持回合中。
- 结果受视频清晰度、帧率、拍摄角度、球场线可见性、遮挡和硬件性能影响。

**后续优化方向**

- 引入更稳定的多回合自动切分。
- 融合姿态、球轨迹和球场区域判断击球归属。
- 增强落点与界内/界外判断。
- 提升 TrackNet/TensorRT 实时性能和跨场景泛化能力。
- 增加更多可视化报表和训练复盘指标。

**建议视觉**

- 使用“已完成 / 进行中 / 未来规划”三列表。
- 最后一行总结：`从视频可视化走向可量化训练复盘`。
