# WFBARNet 项目测试报告

- 报告日期：2026-05-13

## 6. 主要发现

### 6.1 优势

- 自动化测试结果稳定，95 个测试全部通过，说明轨迹后处理、球场检测接口、姿态过滤、回合统计等核心逻辑具备较好的回归基础。
- 姿态输出在真实日志中的非空率达到 93.84%，说明球员检测/姿态链路整体稳定。
- 球场检测一旦被触发，其有效率达到 93.04%，平均置信度 0.8883，说明当前分割式球场检测在命中样本上的质量较高。

### 6.2 风险与不足

- 羽毛球可见率仅为 48.72%，说明轨迹链路仍然是整套系统里最脆弱的环节，容易受高速、小目标、模糊、遮挡和出画影响。
- 57 组运行日志中，只有 3 组实际出现了有效球场检测结果，说明球场模块的真实运行覆盖面仍然偏窄。也就是说，“球场检测质量高”这一结论更多成立于少量已命中样本，而非全部运行场景。
- 运行日志只体现了 `shuttlecourt_seg` 方案，没有覆盖 `opencv` 后端的真实运行证据，因此 OpenCV 球场检测的集成稳定性仍主要依赖单元测试而非现场日志。
- 从日志可推断，球场检测并不是逐帧执行，而更像是按条件触发的稀疏更新流程；否则以约 1.41 s 的平均检测耗时很难满足实时需求。这一策略本身合理，但也意味着需要额外验证“长时间不更新时的漂移风险”。
- 日志样本分布较集中，`MVI_0212` 单一来源占全部帧数的 76.71%，对于不同机位、不同清晰度和不同比赛视频的泛化判断仍不充分。
- 自动化测试尚未覆盖 EXE 打包验证、GUI 主流程联调、摄像头实时处理和长视频连续运行稳定性。

## 7. 结论

综合自动化测试与本地日志来看，WFBARNet 当前版本已经具备较好的算法模块稳定性，尤其是在轨迹事件规则、姿态过滤、回合统计和球场分割检测的“命中样本”上表现较稳，适合继续用于本地调试、算法迭代和实验验证。

但如果目标是对外演示、长期使用或发布桌面端程序，当前版本仍需要补齐三类测试空白：

- 多来源真实视频回归，降低对 `MVI_0212` 单一来源的依赖。
- 球场检测全链路验证，尤其是非分割后端与长视频稀疏更新场景。
- GUI 与 EXE 端到端测试，覆盖打包、启动、视频加载、导出和异常恢复。

## 8. 建议的下一步测试计划

- 增加一个“标准回归样本集”，至少覆盖室内、室外、不同机位、不同清晰度和不同比赛来源。
- 为球场检测增加运行时统计报表，单独记录“尝试次数、成功次数、连续复用时长、漂移修正次数”。
- 补充 PyQt6 主流程的集成测试，至少验证视频加载、分析启动、暂停恢复、结果导出和异常提示。
- 对打包后的 EXE 做一次最小发布验收，记录启动耗时、模型加载耗时和首帧分析耗时。
- 将当前日志统计脚本沉淀为可重复运行的测试工具，避免后续每次手工汇总。
